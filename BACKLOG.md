# 商用向け実装バックログ

## 目的

現在の本システムは、図書 CRUD、Docker 運用、CI、Playwright E2E までを学習用として段階的に整備できている。
一方で、商用プロダクトとして継続運用するための基盤機能は未整備な部分が多い。

このバックログでは、商用利用を見据えて今後追加したい実装を整理し、優先順位と実装順の目安を管理する。

## 現状認識

- 現在の中心機能は図書の CRUD である
- frontend / backend / db / Docker Compose / CI / Playwright E2E の基本導線はある
- 認証、認可、構造化ログ、監査ログ、監視などの運用基盤は未整備である
- 商用化を進めるには、機能追加より先に運用基盤とセキュリティ基盤を固める必要がある

## 直近方針

直近では、非機能要件の強化よりも、まず利用者にとって変化が大きい機能面の改善を優先する。

そのため、本バックログでは次の 2 軸で管理する。

- 第1優先: 利用者体験を大きく改善する機能追加
- 第2優先: 商用運用に必要な非機能要件の強化

非機能要件も将来的には重要だが、直近は「一覧が使いやすい」「目的の本を探しやすい」「1件ずつ確認しやすい」といった機能面の改善から着手する。

## 機能面の優先バックログ

### F1. 検索機能

#### 目的

利用者が大量の本の中から目的の 1 冊を探しやすくする。

#### 追加したい内容

- タイトルでの部分一致検索

#### 期待する効果

- 一覧を目視で追わずに目的の本へたどり着ける
- データ件数が増えたときの使い勝手が大きく改善する

### F2. ページネーション

- 前へ / 次へ のページ移動 UI

#### 期待する効果

### F3. 並び替え

- 昇順 / 降順の切り替え方針

#### 期待する効果

### F4. 本詳細画面

#### 目的

一覧だけでは見づらい 1 冊ごとの情報を、落ち着いて確認できるようにする。

#### 追加したい内容

- 一覧画面から詳細画面への導線

#### 期待する効果

- 一覧画面に情報を詰め込みすぎずに済む

### F5. フォーム UX 改善

### F6. 削除体験の改善

#### 目的

誤削除を減らし、削除操作の意図を利用者が確認しやすくする。

#### 追加したい内容

- 削除対象タイトルの表示

#### 期待する効果

- 重要操作の誤実行を減らせる

### F7. 監査ログ画面

#### 期待する効果

## 優先バックログ

### 1. 構造化ログ

#### 目的

障害調査、運用監視、問い合わせ対応のために、API リクエストごとの情報を機械的に追えるようにする。

#### 追加したい内容

- `timestamp`
- `level`
- `request_id`
- `path`
- `method`
- `status_code`
- `latency_ms`

#### 期待する効果

- 障害時に「いつ」「どの API で」「どの程度遅かったか」を追跡できる
- 後続の認証・監査・監視機能の土台になる

### 2. グローバル例外ハンドリング

#### 目的

例外処理を API ごとに分散させず、利用者向けレスポンスと内部ログ出力を分離して一貫性を持たせる。

#### 追加したい内容

- FastAPI の共通例外ハンドラ
- 想定外例外の `500 Internal Server Error` 統一レスポンス
- 利用者向けメッセージと内部調査用ログの分離
- `request_id` と紐づく障害追跡

#### 期待する効果

- エラー時のレスポンス形式が安定する
- フロントエンド側のエラー表示ルールを整理しやすくなる
- 商用運用時の障害調査がしやすくなる

### 3. 認証

#### 目的

商用運用で前提になる「誰が利用しているか」を判別できる状態を作る。

#### 追加したい内容

- ログイン API
- ログアウト処理
- パスワードハッシュ化
- セッションまたは JWT による認証状態管理
- 認証必須 API の導入

#### 期待する効果

- 未認証利用者の操作を制限できる
- 利用者単位での機能制御や監査につなげられる

### 4. 認可

#### 目的

ログイン済みであることに加えて、「何をしてよいか」を制御できるようにする。

#### 追加したい内容

- ロール設計
- 一般ユーザー / 管理者などの権限分離
- API ごとの操作権限チェック

#### 期待する効果

- 誤操作や不正操作の範囲を制限できる
- 将来的な管理画面や運用者機能へつなげやすい

### 5. 監査ログ

#### 目的

通常のアプリケーションログとは別に、重要操作の履歴を残す。

#### 追加したい内容

- 誰が
- いつ
- どのデータに対して
- 作成 / 更新 / 削除を行ったか

#### 期待する効果

- 重要操作の追跡ができる
- 問い合わせ対応や内部統制に役立つ

### 6. ユーザー管理

#### 目的

認証を継続運用できるように、利用者情報を管理する基盤を作る。

#### 追加したい内容

- `users` テーブル
- 初期管理者の作成方針
- パスワード変更
- ユーザー無効化
- 将来的なメール認証や招待機能の土台

#### 期待する効果

- 商用運用に必要な利用者管理の入口を作れる
- 認証・認可・監査の前提データを整えられる

### 7. 入力・セキュリティ強化

#### 目的

本番運用を見据えて、開発向け設定のままになっている部分を段階的に見直す。

#### 追加したい内容

- CORS 設定の見直し
- 環境別設定の整理
- 本番用 origin 制限
- セキュリティヘッダ
- レート制限
- 依存ライブラリの脆弱性確認

#### 期待する効果

- セキュリティ事故の入口を減らせる
- 本番環境と開発環境の責務を分離しやすくなる

### 8. 監視・ヘルスチェック拡張

#### 目的

アプリが「起動している」だけではなく、「正常に使える」状態かを判定できるようにする。

#### 追加したい内容

- liveness / readiness の分離
- DB 接続確認を含むヘルスチェック
- メトリクス収集
- アラート設計の前提整理

#### 期待する効果

- 障害の早期検知がしやすくなる
- CI / Docker / 将来の本番環境での運用判断に使える

### 9. データ運用改善

#### 目的

削除や更新の扱いを商用運用向けに見直す。

#### 追加したい内容

- `hard delete` のままにするかの見直し
- `soft delete` 導入検討
- 復元要件の整理

#### 期待する効果

- 誤削除時の復旧可能性を高められる
- 業務要件に合わせたデータ保持方針を決めやすくなる

### 10. 一覧 API の商用化

#### 目的

データ件数が増えても扱えるように、一覧取得を拡張する。

#### 追加したい内容

- ページネーション
- 検索
- ソート
- フィルタ

#### 期待する効果

- 件数増加時の性能と操作性を改善できる
- 商用利用時の一覧画面の実用性が上がる

### 11. Batch 実行基盤

#### 目的

定期処理や一括処理を段階的に追加できるようにし、画面操作とは別の運用導線を整える。

#### 追加したい内容

- batch 用ディレクトリ構成
- CLI などの手動実行入口
- service / repository を再利用する実装方針
- 実行結果の記録
- 冪等性を意識した実行設計

#### 期待する効果

- 画面経由では扱いにくい一括処理を安全に追加できる
- 今後の定期実行や運用作業の土台になる

### 12. books 向け Batch 機能

#### 目的

現在の中心機能である `books` を題材に、実務で使いやすい一括入出力処理を追加する。

#### 追加したい内容

- CSV import batch
- CSV export batch
- 一括登録 / 一括更新の方針整理
- 入力エラー行の扱い
- 実行結果サマリーの出力

#### 期待する効果

- 既存の `books` ドメインを使って Batch 処理を学べる
- 図書データの初期投入やメンテナンス作業を効率化できる

## 優先順位の考え方

最初に着手するべきなのは、認証そのものよりも、まず運用時に状況を追跡できる基盤である。
そのため、初手は以下の順を基本方針とする。

1. 構造化ログ
2. グローバル例外ハンドリング
3. 認証
4. 認可
5. 監査ログ

理由は、認証を導入すると失敗パターン、分岐、問い合わせパターンが増えるため、その前にログと例外の基盤を整えておく方が安全だからである。

Batch 処理は運用機能として有用だが、認証・認可・監査の前に広げると、実行主体や失敗時調査のルールが曖昧になりやすい。
そのため、最初の Batch は認可と監査ログの方針が見えた後に入れる。
