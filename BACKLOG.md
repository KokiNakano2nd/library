# 商用向け実装バックログ

## 目的

現在の本システムは、図書 CRUD、Docker 運用、CI、Playwright E2E までを学習用として段階的に整備できている。
一方で、商用プロダクトとして継続運用するための基盤機能は未整備な部分が多い。

このバックログでは、商用利用を見据えて今後追加したい実装を整理し、優先順位と実装順の目安を管理する。

## 現状認識

- 現在の中心機能は図書の CRUD である
- frontend / backend / db / Docker Compose / CI / Playwright E2E の基本導線はある
- 認証、認可、構造化ログ、監査ログ、監視などの運用基盤は未整備である
- 商用化を進めるには、機能追加より先に運用基盤とセキュリティ基盤を固める必要がある

## 優先バックログ

### 1. 構造化ログ

#### 目的

障害調査、運用監視、問い合わせ対応のために、API リクエストごとの情報を機械的に追えるようにする。

#### 追加したい内容

- `timestamp`
- `level`
- `request_id`
- `path`
- `method`
- `status_code`
- `latency_ms`

#### 期待する効果

- 障害時に「いつ」「どの API で」「どの程度遅かったか」を追跡できる
- 後続の認証・監査・監視機能の土台になる

### 2. グローバル例外ハンドリング

#### 目的

例外処理を API ごとに分散させず、利用者向けレスポンスと内部ログ出力を分離して一貫性を持たせる。

#### 追加したい内容

- FastAPI の共通例外ハンドラ
- 想定外例外の `500 Internal Server Error` 統一レスポンス
- 利用者向けメッセージと内部調査用ログの分離
- `request_id` と紐づく障害追跡

#### 期待する効果

- エラー時のレスポンス形式が安定する
- フロントエンド側のエラー表示ルールを整理しやすくなる
- 商用運用時の障害調査がしやすくなる

### 3. 認証

#### 目的

商用運用で前提になる「誰が利用しているか」を判別できる状態を作る。

#### 追加したい内容

- ログイン API
- ログアウト処理
- パスワードハッシュ化
- セッションまたは JWT による認証状態管理
- 認証必須 API の導入

#### 期待する効果

- 未認証利用者の操作を制限できる
- 利用者単位での機能制御や監査につなげられる

### 4. 認可

#### 目的

ログイン済みであることに加えて、「何をしてよいか」を制御できるようにする。

#### 追加したい内容

- ロール設計
- 一般ユーザー / 管理者などの権限分離
- API ごとの操作権限チェック

#### 期待する効果

- 誤操作や不正操作の範囲を制限できる
- 将来的な管理画面や運用者機能へつなげやすい

### 5. 監査ログ

#### 目的

通常のアプリケーションログとは別に、重要操作の履歴を残す。

#### 追加したい内容

- 誰が
- いつ
- どのデータに対して
- 作成 / 更新 / 削除を行ったか

#### 期待する効果

- 重要操作の追跡ができる
- 問い合わせ対応や内部統制に役立つ

### 6. ユーザー管理

#### 目的

認証を継続運用できるように、利用者情報を管理する基盤を作る。

#### 追加したい内容

- `users` テーブル
- 初期管理者の作成方針
- パスワード変更
- ユーザー無効化
- 将来的なメール認証や招待機能の土台

#### 期待する効果

- 商用運用に必要な利用者管理の入口を作れる
- 認証・認可・監査の前提データを整えられる

### 7. 入力・セキュリティ強化

#### 目的

本番運用を見据えて、開発向け設定のままになっている部分を段階的に見直す。

#### 追加したい内容

- CORS 設定の見直し
- 環境別設定の整理
- 本番用 origin 制限
- セキュリティヘッダ
- レート制限
- 依存ライブラリの脆弱性確認

#### 期待する効果

- セキュリティ事故の入口を減らせる
- 本番環境と開発環境の責務を分離しやすくなる

### 8. 監視・ヘルスチェック拡張

#### 目的

アプリが「起動している」だけではなく、「正常に使える」状態かを判定できるようにする。

#### 追加したい内容

- liveness / readiness の分離
- DB 接続確認を含むヘルスチェック
- メトリクス収集
- アラート設計の前提整理

#### 期待する効果

- 障害の早期検知がしやすくなる
- CI / Docker / 将来の本番環境での運用判断に使える

### 9. データ運用改善

#### 目的

削除や更新の扱いを商用運用向けに見直す。

#### 追加したい内容

- `hard delete` のままにするかの見直し
- `soft delete` 導入検討
- 復元要件の整理

#### 期待する効果

- 誤削除時の復旧可能性を高められる
- 業務要件に合わせたデータ保持方針を決めやすくなる

### 10. 一覧 API の商用化

#### 目的

データ件数が増えても扱えるように、一覧取得を拡張する。

#### 追加したい内容

- ページネーション
- 検索
- ソート
- フィルタ

#### 期待する効果

- 件数増加時の性能と操作性を改善できる
- 商用利用時の一覧画面の実用性が上がる

### 11. Batch 実行基盤

#### 目的

定期処理や一括処理を段階的に追加できるようにし、画面操作とは別の運用導線を整える。

#### 追加したい内容

- batch 用ディレクトリ構成
- CLI などの手動実行入口
- service / repository を再利用する実装方針
- 実行結果の記録
- 冪等性を意識した実行設計

#### 期待する効果

- 画面経由では扱いにくい一括処理を安全に追加できる
- 今後の定期実行や運用作業の土台になる

### 12. books 向け Batch 機能

#### 目的

現在の中心機能である `books` を題材に、実務で使いやすい一括入出力処理を追加する。

#### 追加したい内容

- CSV import batch
- CSV export batch
- 一括登録 / 一括更新の方針整理
- 入力エラー行の扱い
- 実行結果サマリーの出力

#### 期待する効果

- 既存の `books` ドメインを使って Batch 処理を学べる
- 図書データの初期投入やメンテナンス作業を効率化できる

## 優先順位の考え方

最初に着手するべきなのは、認証そのものよりも、まず運用時に状況を追跡できる基盤である。
そのため、初手は以下の順を基本方針とする。

1. 構造化ログ
2. グローバル例外ハンドリング
3. 認証
4. 認可
5. 監査ログ

理由は、認証を導入すると失敗パターン、分岐、問い合わせパターンが増えるため、その前にログと例外の基盤を整えておく方が安全だからである。

Batch 処理は運用機能として有用だが、認証・認可・監査の前に広げると、実行主体や失敗時調査のルールが曖昧になりやすい。
そのため、最初の Batch は認可と監査ログの方針が見えた後に入れる。

## 推奨実装順

### Step 24: 構造化ログ + request ID + 例外ハンドラ

- API ごとのログを残せるようにする
- `request_id` をレスポンスとログへ載せる
- 想定外例外を共通ハンドラで処理する

### Step 25: users テーブル + パスワードハッシュ

- 利用者情報の保存先を追加する
- 平文パスワードを保存しない構成にする

### Step 26: ログイン API + 認証状態管理

- ログイン処理を追加する
- セッションまたは JWT による認証状態を扱えるようにする

### Step 27: 認可 + 認証必須化

- ロールに応じた制御を追加する
- 図書 API のうち保護すべき操作を認証必須にする

### Step 28: 監査ログ

- 作成、更新、削除の操作履歴を残す
- 利用者と変更対象を追跡できるようにする

### Step 29: Batch 実行基盤

- batch 用の実行入口を追加する
- 実行結果、成功件数、失敗件数を追跡できるようにする
- 再実行に耐えやすい構成を決める

### Step 30: books CSV import / export batch

- `books` を対象に CSV import / export を追加する
- 一括登録時の入力検証とエラー行の扱いを決める
- 実行結果を確認できるようにする

### Step 31: ページネーション / 検索

- 商用利用を想定した一覧 API の拡張を行う

### Step 32: レート制限と本番設定強化

- 本番運用向けの安全性を高める

## 次の候補

現時点で、次に着手する候補は以下の 2 つである。

1. 堅実案: 構造化ログと例外統一
2. 目玉機能案: 認証基盤の導入

商用プロダクトレベルを優先する場合は、まず `構造化ログ + 例外統一` から着手する方針を推奨する。
