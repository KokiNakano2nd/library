# 学習優先度メモ

## 目的

Docker 対応の完了後に、次に何を学習・実装するかを優先順で整理するためのメモです。
現時点では `LEARNING_PROGRESS.md` の内容を基準に、`Step 16` は完了、次は `Step 17` から進める前提でまとめます。

## 現在地

- `Step 16: Docker上での疎通確認` は完了
- 次に行うことは `Step 17: Playwright による Docker 環境の E2E 確認`
- その後に `Step 18: Docker化内容のドキュメント反映`

## 優先順位

### 優先度1: Step 17 Playwright による Docker 環境の E2E 確認

#### ねらい

Docker 化したことで画面遷移や CRUD が壊れていないかを、ブラウザ操作込みで確認する。

#### この段階でやること

- Docker Compose で `frontend` `backend` `db` を起動する
- Playwright で本の作成、一覧表示、編集、削除を確認する
- 証跡を `test/evidence` 配下に保存する
- Docker 環境で発生したエラーがあれば切り分ける

#### なぜ優先度が高いか

- Docker 化の価値は「起動できる」だけでなく「既存機能が壊れていない」と確認できて初めて出るため
- この確認が終わると、次の CI で何を自動化するかが明確になるため

### 優先度2: Step 18 Docker化内容のドキュメント反映

#### ねらい

実装内容と学習記録を一致させ、README を SSoT とする運用を守る。

#### この段階でやること

- `README.md` に Docker 前提の仕様を反映する
- `ELPLANATION/EXPLANATION_STEP17.md` と `ELPLANATION/EXPLANATION_STEP18.md` を必要に応じて作成・更新する
- `LEARNING_PROGRESS.md` のステータス、理解度、日付、学びを更新する
- 必要であれば `LEARNING_ROADMAP.md` の完了状況も更新する

#### なぜ優先度が高いか

- このプロジェクトでは README が仕様の SSoT だから
- 後から CI を入れるときに、手順や前提条件が文書化されていると迷いが減るため

### 優先度3: CI の導入

#### ねらい

今まで手動で行っていた確認を、自動で毎回実行できるようにする。

#### 最初に狙う範囲

- backend の `ruff` と `pytest`
- frontend の `eslint` と `build`
- Docker Compose 起動後の Playwright 実行

#### 進め方の方針

- 最初は CD まで広げず、CI のみを小さく導入する
- 1本の workflow ですべて詰め込まず、lint/test/e2e を段階的に追加する
- ローカルで成功しているコマンドをそのまま CI に移植する

#### なぜこの順番か

- Step 17 と Step 18 が終わると、自動化対象と期待結果が明確になるため
- 先に CI を入れても、Docker 環境の確認観点が固まっていないと workflow が不安定になりやすいため

### 優先度4: CD またはデプロイ準備

#### 候補

- コンテナイメージの build と push
- ステージング環境への手動デプロイ
- 本番公開を想定した環境変数整理

#### 位置づけ

- 学習題材としては有効
- ただし現段階では、CI を安定させてから着手するほうが理解しやすい

### 優先度5: 周辺改善

#### 候補

- 開発用 PowerShell スクリプトの整理
- `.env` の責務整理
- ログ確認方法の明確化
- エラー時の切り分け手順の整理

#### 位置づけ

- 実務では重要
- ただし今すぐでなくても、Step 17-18 と CI の途中で必要になった時点で小さく追加すればよい

## おすすめの進め方

1. `Step 17` を完了する
2. `Step 18` で README と学習記録を揃える
3. CI を導入して、手動確認を自動化する
4. その後に CD やデプロイ準備へ進む

## ひとことで言うと

次に最も自然なのは `Docker環境のE2E確認 -> ドキュメント反映 -> CI導入` の順です。
特に CI は有力な次候補だが、先に Docker 環境での確認観点を固めてから入れるほうが、学習効果と実務的な整理の両方で筋が良いです。
